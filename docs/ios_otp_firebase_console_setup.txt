 iOS Phone Authentication (OTP) — Firebase Console Setup Guide

This document lists the manual steps that MUST be done in the Firebase Console
and the Apple Developer portal so that the iOS app verification flow shipped on
the `ios-otp` branch actually works in production.

The code-side changes (AppDelegate, Info.plist URL scheme, Runner.entitlements,
Xcode build settings) are already applied on the branch. Everything below is
infrastructure / portal configuration that cannot be committed to source.


--------------------------------------------------------------------------------
 1. APPLE DEVELOPER PORTAL — CREATE AN APNS AUTHENTICATION KEY (.p8)
--------------------------------------------------------------------------------

Firebase Phone Auth verifies iOS devices primarily through a silent APNs push.
Without a valid APNs key, every verification falls back to reCAPTCHA, which is
slower and more error-prone.

  1.1  Sign in at https://developer.apple.com/account
  1.2  Go to: Certificates, Identifiers & Profiles  →  Keys
  1.3  Click the "+" button to register a new key.
  1.4  Name it something recognizable, e.g. "Aban Sermon — APNs Auth Key".
  1.5  Check the box for "Apple Push Notifications service (APNs)".
  1.6  Click Continue, then Register.
  1.7  DOWNLOAD the .p8 file IMMEDIATELY. Apple only lets you download it ONCE.
       Store it in a secure location (1Password / shared vault).
  1.8  Note down the following — you will need them in step 2:
         - Key ID    (10 characters, shown next to the key name)
         - Team ID   (top-right of the developer portal, 10 characters)


--------------------------------------------------------------------------------
 2. APPLE DEVELOPER PORTAL — ENABLE PUSH NOTIFICATIONS ON THE APP ID
--------------------------------------------------------------------------------

  2.1  Certificates, Identifiers & Profiles  →  Identifiers
  2.2  Locate the App ID for `com.example.abansermon`.
       (If it does not exist, create it: type = App, Bundle ID = explicit.)
  2.3  Click the App ID, scroll to "Capabilities".
  2.4  Tick "Push Notifications" and Save.
  2.5  If you use provisioning profiles manually (not Automatic Signing),
       regenerate the Development AND Distribution profiles for this App ID
       AFTER enabling Push Notifications, otherwise builds will fail to sign.


--------------------------------------------------------------------------------
 3. FIREBASE CONSOLE — UPLOAD THE APNS KEY
--------------------------------------------------------------------------------

  3.1  Open https://console.firebase.google.com/project/aban-sermon
  3.2  Project Settings (gear icon, top-left)  →  "Cloud Messaging" tab.
  3.3  Scroll down to: "Apple app configuration"  →  the iOS app row
       (com.example.abansermon).
  3.4  Under "APNs Authentication Key", click "Upload".
  3.5  Choose the .p8 file downloaded in step 1.7.
  3.6  Enter the Key ID and Team ID noted in step 1.8.
  3.7  Click Upload.

  NOTE: This single key is used for BOTH FCM (firebase_messaging) and Firebase
  Phone Auth's silent-push verification — you do NOT need a separate key.


--------------------------------------------------------------------------------
 4. FIREBASE CONSOLE — ENABLE THE PHONE PROVIDER
--------------------------------------------------------------------------------

  4.1  Build  →  Authentication  →  Sign-in method.
  4.2  In the "Sign-in providers" list, click "Phone".
  4.3  Toggle "Enable" ON, then Save.
  4.4  (Optional but recommended for development)
       Expand "Phone numbers for testing".
       Add at least one fictional number + a fixed code so the team can test
       login without burning real SMS quota and without needing a working
       APNs setup. Example:
         Phone:  +966500000001        Code:  123456
         Phone:  +966500000002        Code:  654321
       These numbers will NOT trigger APNs / reCAPTCHA — Firebase short-circuits
       them and returns the configured code directly.


--------------------------------------------------------------------------------
 5. FIREBASE CONSOLE — VERIFY THE iOS APP REGISTRATION
--------------------------------------------------------------------------------

  5.1  Project Settings  →  General tab  →  "Your apps".
  5.2  Confirm that the iOS app entry shows:
         - Bundle ID:    com.example.abansermon
         - App ID:       1:688102645842:ios:f4fd83753c060474433f8b
  5.3  If the bundle ID was changed (recommended before going to App Store —
       `com.example.*` will be rejected by Apple), do the following:
         a) Click "Add app" → iOS, register the new bundle ID.
         b) Download the new GoogleService-Info.plist.
         c) Replace `ios/Runner/GoogleService-Info.plist` in the repo.
         d) Update the URL scheme in `ios/Runner/Info.plist` — replace the
            `app-1-688102645842-ios-...` scheme with the new app's encoded ID
            (format: `app-1-<PROJECT_NUMBER>-ios-<APP_ID_SUFFIX>`).
         e) Update `lib/firebase_options.dart` (run `flutterfire configure`).


--------------------------------------------------------------------------------
 6. (RECOMMENDED) FIREBASE APP CHECK — DEVICECHECK / APP ATTEST
--------------------------------------------------------------------------------

App Check protects the Phone Auth endpoint from abuse (toll fraud via mass
verification requests). It is OFF by default. Recommended before public launch.

  6.1  Build  →  App Check  →  Apps tab.
  6.2  Click the iOS app row  →  "Register".
  6.3  Choose a provider:
         - "App Attest" for iOS 14+ (recommended, no extra config).
         - "DeviceCheck" if you must support iOS 11–13 (requires a Team ID +
           Key ID + private key, similar to the APNs key).
  6.4  Save.
  6.5  Once you are confident the app works WITH App Check in monitoring mode,
       go to: App Check  →  APIs tab  →  Authentication  →  set to "Enforced".

  NOTE: When enforcement is ON, ALL phone verification calls from clients that
  do NOT pass App Check will be rejected. Roll out monitoring-only first, watch
  the metrics for at least a week, and only flip to Enforced once the failure
  rate is < 1%.


--------------------------------------------------------------------------------
 7. (OPTIONAL) BILLING — SMS QUOTA
--------------------------------------------------------------------------------

  7.1  Phone Auth has a free daily SMS quota. Production traffic will exceed it.
  7.2  Project Settings  →  Usage and billing  →  upgrade the project to the
       Blaze (pay-as-you-go) plan if not already.
  7.3  Set a budget alert (Google Cloud Console  →  Billing  →  Budgets) so the
       team is paged before a runaway loop drains the SMS budget.


--------------------------------------------------------------------------------
 8. XCODE — ENABLE THE PUSH NOTIFICATIONS CAPABILITY (one-time, on a Mac)
--------------------------------------------------------------------------------

The repo already contains `ios/Runner/Runner.entitlements` with `aps-environment
= development`, and the Xcode project is wired to use it via
`CODE_SIGN_ENTITLEMENTS`. However, the Apple-side capability still has to be
flipped on for the target so that signing matches the entitlement:

  8.1  Open `ios/Runner.xcworkspace` in Xcode (NOT the .xcodeproj).
  8.2  Select the Runner target  →  "Signing & Capabilities" tab.
  8.3  Pick the correct Team. Confirm "Automatically manage signing" is ON.
  8.4  Click "+ Capability"  →  add "Push Notifications".
  8.5  Click "+ Capability"  →  add "Background Modes"  →  tick:
         [x] Remote notifications
       (Required so that Firebase Auth's silent push is delivered while the
        app is backgrounded.)
  8.6  Build to a real device (NOT the simulator — APNs does not work on
       the iOS simulator below iOS 16, and even on 16+ requires a paid
       Apple Developer account signed in to Xcode).


--------------------------------------------------------------------------------
 9. END-TO-END TEST CHECKLIST
--------------------------------------------------------------------------------

  [ ] APNs key uploaded to Firebase (step 3).
  [ ] Phone provider enabled in Firebase Auth (step 4).
  [ ] Test phone numbers added (step 4.4) — verify login with one of them.
  [ ] Push Notifications + Background Modes enabled in Xcode (step 8).
  [ ] App installed on a REAL iOS device, signed with the matching team.
  [ ] Request OTP for a real Saudi number (+9665XXXXXXXX). SMS arrives.
  [ ] Enter the 6-digit code in the OTP screen. Login succeeds, navigates
      to /auth-success.
  [ ] Re-test in airplane mode + Wi-Fi only (so APNs may fail) — confirm
      the reCAPTCHA fallback opens a Safari sheet, completes, and returns
      to the app via the `app-1-688102645842-ios-...` URL scheme.
  [ ] Production build (Release configuration) signed with a Distribution
      profile — the Runner.entitlements `aps-environment` will need to be
      flipped to `production` for App Store / TestFlight builds. This can
      be handled by adding a Release-only entitlements file later, but the
      single `development` entry is fine for development + ad-hoc builds.


--------------------------------------------------------------------------------
 10. WHAT WAS CHANGED IN CODE (for reviewer reference)
--------------------------------------------------------------------------------

  - ios/Runner/AppDelegate.swift
      * Imports FirebaseAuth + UserNotifications.
      * Registers for remote notifications at launch (so APNs token reaches
        Firebase Auth via the default app-delegate proxy).
      * Forwards URL-open events to Auth.auth().canHandle(url) so the
        reCAPTCHA fallback redirect is consumed by Firebase Auth.

  - ios/Runner/Info.plist
      * Added CFBundleURLTypes with the encoded App ID
        (app-1-688102645842-ios-f4fd83753c060474433f8b) as a URL scheme,
        used by reCAPTCHA fallback to redirect back into the app.

  - ios/Runner/Runner.entitlements   (NEW)
      * aps-environment = development.

  - ios/Runner.xcodeproj/project.pbxproj
      * CODE_SIGN_ENTITLEMENTS = Runner/Runner.entitlements; added to the
        Debug, Release and Profile configurations of the Runner target.
